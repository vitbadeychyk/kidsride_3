KidsRide — статичний сайт + Vercel API + Supabase
====================================================

ЯК РОЗГОРНУТИ:
1. Розпакуйте архів — усі файли мають бути в КОРЕНІ репозиторію
   (не в підпапці!).
2. Залийте вміст у GitHub-репозиторій.
3. На Vercel: New Project → імпортуйте репозиторій → Framework
   Preset: "Other" → Root Directory: ./ → Deploy.
   Жодних build-команд не потрібно — це чистий статичний сайт
   + одна serverless-функція /api/lead.js (Vercel її підхопить
   автоматично з папки api/).

ГОЛОВНЕ:
- index.html має бути в корені (не в /public/, не в /artifacts/)
- shared.js теж у корені — звідти його завантажують усі сторінки.
- api/lead.js — серверна функція Vercel для прийому заявок.

ОНОВЛЕННЯ 1 — "Під замовлення"
═══════════════════════════════════════════════════════════════
Додано колонку "Під замовлення" (днів готовності) для товарів.
Якщо залишок 0, але "Під замовлення" > 0 — товар показується
в каталозі з підписом "Під замовлення X днів" (синім).
Якщо залишок = 1, картка показує зелене "В наявності" + значок
"остання модель".

SQL у Supabase (SQL Editor → New query → Run):

  ALTER TABLE products
  ADD COLUMN IF NOT EXISTS preorder_days int NOT NULL DEFAULT 0;

ОНОВЛЕННЯ 2 — "Купити в 1 клік" + Заявки в адмінці + Telegram
═══════════════════════════════════════════════════════════════

Як це працює:
  Користувач натискає "Купити в 1 клік" → вводить телефон →
  заявка одночасно (а) зберігається в Supabase (таблиця leads),
  і (б) приходить вам у Telegram моментально.
  В адмінці є вкладка "Заявки" зі списком, статусами і пошуком.

ШАГ 1. Створіть таблицю leads у Supabase (SQL Editor):

  CREATE TABLE IF NOT EXISTS leads (
    id            BIGSERIAL PRIMARY KEY,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    phone         TEXT NOT NULL,
    name          TEXT,
    product_id    TEXT,
    product_name  TEXT,
    product_price NUMERIC,
    product_brand TEXT,
    source        TEXT DEFAULT 'quick_order',
    status        TEXT NOT NULL DEFAULT 'new',
    comment       TEXT,
    items         JSONB,
    notes         TEXT
  );
  CREATE INDEX IF NOT EXISTS leads_created_idx ON leads(created_at DESC);
  CREATE INDEX IF NOT EXISTS leads_status_idx  ON leads(status);

  -- Дозволити адмінці читати/змінювати через anon-ключ
  ALTER TABLE leads ENABLE ROW LEVEL SECURITY;
  CREATE POLICY leads_all ON leads FOR ALL TO anon, authenticated USING (true) WITH CHECK (true);

  -- Якщо у вас УЖЕ є політика leads_all (TO anon) і заявки в адмінці не видно,
  -- виконайте:
  --   DROP POLICY IF EXISTS leads_all ON leads;
  --   CREATE POLICY leads_all ON leads FOR ALL TO anon, authenticated USING (true) WITH CHECK (true);

ШАГ 2. Створіть Telegram-бота:
  1) Відкрийте в Telegram @BotFather → /newbot
  2) Дайте боту назву (наприклад "KidsRide Заявки")
  3) Дайте йому username (має закінчуватись на "bot",
     наприклад "kidsride_leads_bot")
  4) BotFather дасть TOKEN (виглядає як 123456:ABC-DEF...).
     Збережіть його — це TELEGRAM_BOT_TOKEN.

ШАГ 3. Дізнайтесь свій chat_id:
  1) Знайдіть у Telegram свого бота і натисніть /start
  2) Відкрийте у браузері (підставте свій токен):
     https://api.telegram.org/bot<TOKEN>/getUpdates
  3) Знайдіть "chat":{"id":123456789, ... — це і є ваш chat_id.
  4) Якщо хочете отримувати в групу — додайте бота в групу,
     напишіть будь-яке повідомлення в неї і знову відкрийте
     getUpdates. У групи chat_id буде з мінусом, наприклад
     -1001234567890.

ШАГ 4. Дізнайтесь Service Role ключ Supabase (рекомендовано):
  Supabase → Project Settings → API → "service_role" secret.
  УВАГА: це секретний ключ — його НЕ можна викладати у фронтенд.
  У нас він буде тільки на сервері Vercel. Якщо не хочете його
  давати — пропустіть, тоді функція використає публічний anon-ключ
  (працюватиме поки RLS-политика дозволяє INSERT для anon).

ШАГ 5. Додайте змінні середовища у Vercel:
  Vercel → ваш проект → Settings → Environment Variables.
  Додайте 4 змінні (для Production, Preview, Development):

    SUPABASE_URL              = https://xczrzdbikkycgpnvolib.supabase.co
    SUPABASE_SERVICE_ROLE_KEY = <ключ із Шагу 4>     (опціонально)
    SUPABASE_ANON_KEY         = <ваш anon-ключ>       (якщо без service_role)
    TELEGRAM_BOT_TOKEN        = <токен із Шагу 2>
    TELEGRAM_CHAT_ID          = <id із Шагу 3>

  Після додавання — натисніть Deploy → Redeploy, щоб змінні
  підхопились.

ШАГ 6. Перевірка:
  1) Відкрийте сайт, натисніть "Купити в 1 клік" на товарі,
     введіть свій реальний телефон.
  2) В Telegram має одразу прийти повідомлення.
  3) В адмінці відкрийте "Заявки" — заявка має бути там зі
     статусом "Нова". Можна змінити статус на
     "Передзвонили / Оформлено / Скасовано".

ЯКЩО ЩОСЬ НЕ ПРАЦЮЄ:
  - Заявка не приходить в Telegram → Vercel → Deployments →
    останній деплой → Functions → /api/lead → Logs. Подивіться
    помилку. Найчастіше — невірний chat_id або токен.
  - Заявка не зберігається в Supabase → перевірте, що виконали
    SQL із Шагу 1, і що змінні SUPABASE_URL та ключ задані.
  - Адмінка показує "Таблиця leads ще не створена" → виконайте
    SQL із Шагу 1.

ОНОВЛЕННЯ 3 — Власний серверний проксі для Нової Пошти
═══════════════════════════════════════════════════════════════

Що змінилось:
  Раніше checkout.html ходив напряму у https://api.novaposhta.ua
  з ключем у JS-коді — це і ліміти, і ризик "засвітити" ключ.
  Тепер додано Vercel-функцію /api/np.js, яка проксює запити
  до Нової Пошти. Ключ зберігається тільки на сервері в env
  NP_API_KEY. На фронтенді ключа більше немає.

Що зробити:

ШАГ 1. Отримайте власний API-ключ Нової Пошти:
  https://new.novaposhta.ua/dashboard/settings/developers
  (Особистий кабінет → Налаштування → Безпека → Ключі API).
  Створіть новий ключ — скопіюйте 32-символьний рядок.

ШАГ 2. Додайте змінну середовища у Vercel:
  Vercel → ваш проект → Settings → Environment Variables →
  Add New (для Production, Preview, Development):

    NP_API_KEY = <ваш 32-символьний ключ із Шагу 1>

  Після додавання — Deployments → останній → Redeploy,
  щоб змінна підхопилась.

ШАГ 3. Перевірка:
  1) Відкрийте сторінку checkout.html на сайті.
  2) Почніть вводити назву міста (від 2 символів) — має зʼявитись
     повний живий список міст з Нової Пошти (а не 20 mock-міст).
  3) Оберіть місто — у полі відділення мають підтягнутись
     всі реальні відділення/поштомати цього міста.

ЯКЩО НЕ ПРАЦЮЄ:
  - У dropdown показуються тільки 20 міст (Київ, Харків, Одеса…) →
    це fallback на mock. Значить /api/np повернув помилку. Перевірте:
    Vercel → Deployments → останній → Functions → /api/np → Logs.
  - "NP_API_KEY не задано" → не додали змінну середовища або
    забули зробити Redeploy після додавання.
  - "Метод X.Y не дозволений" → у /api/np.js є білий список
    дозволених методів (Address.getCities, AddressGeneral.getWarehouses
    тощо). Якщо потрібен ще якийсь метод НП — додайте його в
    об'єкт ALLOWED у файлі api/np.js.

ЩО САМЕ ЗМІНЕНО У КОДІ:
  - Додано:    api/np.js (новий serverless-проксі до НП)
  - Змінено:   checkout.html — замість прямого виклику
               api.novaposhta.ua тепер виклик /api/np,
               без ключа у фронтенді.

ОНОВЛЕННЯ 5 — Поле "Відео" у товарі (YouTube)
═══════════════════════════════════════════════════════════════

В адмінці у формі товару є поле "Відео огляд (URL)".
Підтримуються:
  - https://youtu.be/XXXXXXXXXXX
  - https://www.youtube.com/watch?v=XXXXXXXXXXX
  - https://www.youtube.com/embed/XXXXXXXXXXX
  - https://www.youtube.com/shorts/XXXXXXXXXXX
  - будь-який інший прямий URL — покажеться кнопкою "Відкрити відео"

ОБОВ'ЯЗКОВО додайте колонку в Supabase (інакше поле не збережеться):

  ALTER TABLE products
  ADD COLUMN IF NOT EXISTS video_url text;

Як це працює:
  - При збереженні товару адмінка пише video_url у Supabase.
  - На сторінці product.html?id=<UUID> завантажується це поле і
    вгорі вкладки "Опис" з'являється блок "Відеоогляд" з YouTube-плеєром.
  - Якщо поле порожнє — блок просто не показується.

Зауваження:
  - Якщо колонки ще немає в БД, sbSaveProduct автоматично спробує
    зберегти товар БЕЗ video_url (щоб не блокувати інші зміни) і
    напише warning у консоль. Тому товар збережеться, але відео не
    запишеться, доки ви не виконаєте ALTER TABLE вище.

ОНОВЛЕННЯ 5 — Групування кольорів через КОД МОДЕЛІ (v3.5)
═══════════════════════════════════════════════════════════════

ПРОБЛЕМА:
Раніше варіанти кольору тягнулись за префіксом артикула (до "-N"),
тому, наприклад, "M 4259EBLR-3" і "M 4259EBLRS-3" вважалися різними
товарами і не показувались у одному ряду кольорів.

РІШЕННЯ:
Додано нове поле "Код моделі для кольорів" (model_code) в адмін-панелі.
Якщо у двох товарах вписати один і той самий код (напр. "4259") —
вони автоматично об'єднаються в один блок кольорових варіантів.

ШАГ 1. ОБОВ'ЯЗКОВО додайте колонку в Supabase (SQL Editor → New query → Run):

  ALTER TABLE products
  ADD COLUMN IF NOT EXISTS model_code text;

  CREATE INDEX IF NOT EXISTS products_model_code_idx
    ON products(model_code) WHERE model_code IS NOT NULL;

ШАГ 2. У адмін-панелі відкрийте кожен товар, який має варіанти кольору,
       і впишіть у поле "Код моделі для кольорів" один і той самий код.
       Наприклад для всіх товарів сім'ї Гелік M 4259:
         M 4259EBLR-1   →  4259
         M 4259EBLR-2   →  4259
         M 4259EBLR-3   →  4259
         M 4259EBLRS-3  →  4259

ШАГ 3. Перейдіть на сторінку будь-якого з цих товарів — у блоці
       "В інших кольорах" будуть всі 4 варіанти у єдиному рядку.

ЯК ЦЕ ПРАЦЮЄ ВНУТРІШНЬО:
  - Якщо model_code заповнений — фронт робить запит
    products?model_code=eq.4259  і збирає всі варіанти.
  - Якщо model_code порожній (старі товари) — fallback на старий
    спосіб: пошук за префіксом артикула. Тобто СТАРІ товари
    продовжать працювати без змін, поки ви їм не проставите код.
